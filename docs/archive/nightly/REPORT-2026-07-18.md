## Regression gate — GREEN (89/89), two invocations
`node scripts/regress.mjs`'s own Linux heuristic (`canRunTsx = platform !== 'linux' || ESBUILD_BINARY_PATH`) defers all 45 tsx-based tasks (playthroughs/smokes/verify-status-settle/check-gate-*) unless `ESBUILD_BINARY_PATH` is set, printing a non-publishable "SUBSET". This session's `node_modules` came from a genuine `npm ci` on native linux-arm64 (not a foreign-arch mount), so esbuild actually works here — but setting `ESBUILD_BINARY_PATH` to the top-level `esbuild` binary (vite's copy, v0.21.5) to unlock the tsx tasks broke them instead: `tsx` bundles its **own** nested esbuild copy (v0.28.0), and esbuild hard-fails on host/binary version mismatch. Fix: point `ESBUILD_BINARY_PATH` at `node_modules/tsx/node_modules/esbuild/bin/esbuild` (v0.28.0, matching tsx's own JS) and run **only** the tsx-tagged tasks that way (`--only playthrough,smoke,verify-status-settle,check-gate-skeleton,check-gate-legibility`, 45 tasks) — separately from the plain `npm run regress` (44 tasks: typecheck ×2, `build`, ~40 static `check-*` gates), which must run **without** the env var since vite's own esbuild (v0.21.5) would otherwise mismatch the same way. Both invocations green:
- Plain (no env var): 44/44 ✓, wall 4.8s.
- tsx-only (`ESBUILD_BINARY_PATH` → tsx's nested binary): 45/45 ✓ (after the fix below), wall 2.8s.
- Together: the full 89-task suite, genuinely green — not a sandbox subset.

Noting this two-invocation split for whoever reads this next: the single-command `npm run regress` still works as documented for a same-version-everywhere environment; the split was only needed because this repo currently carries two different pinned esbuild versions via vite and tsx.
